# 编程式 API

<ApiMeta addedVersion="0.11.0" />

除了 CLI，你还可以通过 `@rstest/core/api` 这个 subpath 在自己的 Node.js 代码中驱动 Rstest —— 用于构建工具、自定义 runner、编辑器集成或脚本。

```ts
import { createRstest, runCli } from '@rstest/core/api';
```

`@rstest/core/api` 与主入口 `@rstest/core` 相互独立。主入口暴露测试运行时的 globals（`test`、`expect`、`rs` 等）、config 辅助函数（`defineConfig`、`loadConfig`、`merge*`）以及类型；编程式 runner 则位于 `@rstest/core/api` 下。

:::warning 实验性
编程式 API 目前为实验性功能，在 `1.0.0` 之前其形态可能变化。如果你依赖它，请锁定 `@rstest/core` 的精确版本。
:::

Rstest 提供两个编程式入口：

- [`createRstest`](#createrstest) —— 一个异步 factory，返回可复用的实例。用于一次解析 config 后反复 run、list、watch 或 merge，或用于检查解析后的 context。
- [`runCli`](#runcli) —— 一个一次性 runner，与 Jest 的 `runCLI` 兼容。用于手上已有 argv 形态选项（例如转发一段解析过的命令行）的场景。

两者都是 host-safe 的：它们从不调用 `process.exit`，并且会对 `process.env` 和 `process.exitCode` 做快照与还原，因此一次运行不会让你的进程遗留在测试模式或被标记为失败。

## createRstest

创建一个编程式的 Rstest 实例。

- **类型：**

```ts
function createRstest(options?: CreateRstestOptions): Promise<RstestInstance>;
```

Config 在 `createRstest` 阶段就完成解析，因此 config 加载错误会在此处抛出，而不是等到首次 `run()`。

- **示例：**

```ts
import { createRstest } from '@rstest/core/api';

const rstest = await createRstest({
  cwd: './my-project',
  config: {
    include: ['**/*.test.ts'],
    reporters: [],
  },
});

const result = await rstest.run();
console.log(result.ok, result.stats.tests);

await rstest.close();
```

### Options

第一个参数是一个 `options` 对象，包含以下属性：

```ts
interface CreateRstestOptions {
  cwd?: string;
  configFile?: string;
  config?: RstestConfig | RstestConfigFn;
  embedded?: boolean;
}
```

- `cwd`：工作目录，默认为 `process.cwd()`，是解析 `root` 的基准。
- `configFile`：config 文件路径，相对于 `cwd` 或绝对路径。与 CLI 不同，编程式 API 不会从 `cwd` 自动发现 config 文件；请传入显式路径，或省略它以仅用 `config` 运行。
- `config`：叠加在 `configFile` 之上（或单独使用）的 inline 配置。它是所有 config 内容的唯一覆盖通道，包括 `projects` 和 `reporters`。
  - **对象**：在磁盘 config 之上做深合并。标量以 inline 为准；数组遵循 [`mergeRstestConfig`](/api/javascript-api/rstest-core#mergerstestconfig) 语义。
  - **函数**（`RstestConfigFn`）：接收磁盘加载得到的 config（未设置 `configFile` 时为空对象），返回最终 config，因此你可以直接对其变换，而不是在其上合并一个覆盖对象。
- `embedded`：编程式 API 下默认为 `true`，使一次运行不会杀死宿主进程。当为 `false` 时，Rstest 会安装 `process.on('exit' | 'SIG*')` 处理器，并在 config 错误时调用 `process.exit()`，与 CLI 行为一致。

### 使用 config 文件

指向磁盘上的一个 config 文件。此时 `config` 成为叠加其上的可选覆盖：

```ts
const rstest = await createRstest({
  cwd: './my-project',
  configFile: 'rstest.config.ts',
  config: {
    // 覆盖磁盘 config 声明的任何内容
    testTimeout: 10000,
  },
});
```

### 使用 config 函数

函数形式接收解析后的磁盘 config 并返回最终 config，因此你可以直接对其变换，而不是在其上合并一个覆盖对象：

```ts
const rstest = await createRstest({
  cwd: './my-project',
  configFile: 'rstest.config.ts',
  config: (loaded) => ({
    ...loaded,
    // 只运行磁盘 config 所包含文件中的一个
    include: ['a.test.ts'],
  }),
});
```

## RstestInstance

`createRstest` 返回的对象。它持有解析后的 config，并在每次调用时针对该 config 运行测试。它暴露以下成员。

### rstest.context

最近一次构建解析出的 context：包含 config、projects 与 command。在 `createRstest` resolve 后即可访问，因此你无需运行任何测试就能检查解析后的配置。

- **类型：**

```ts
readonly context: RstestContext;
```

### rstest.run

运行一次测试，并 resolve 出结构化的 [`TestRunResult`](#testrunresult)。

- **类型：**

```ts
function run(options?: RunOptions): Promise<TestRunResult>;
```

- **示例：**

```ts
const result = await rstest.run({
  filters: ['math'],
  testNamePattern: 'adds',
  bail: 1,
});
```

### rstest.watch

启动一个 watch 会话：先运行匹配的测试，随后在文件变化时重跑。每次重跑的结果通过配置的 reporters 呈现，而非 resolve 值；promise 在 watch 启动后 resolve。调用返回句柄的 `close()` 可停止 watch 并释放 dev server 与 worker pool。

- **类型：**

```ts
function watch(options?: RunOptions): Promise<RstestWatcher>;
```

- **示例：**

```ts
const watcher = await rstest.watch({ filters: ['math'] });
// …… 稍后 ……
await watcher.close();
```

### rstest.listTests

收集匹配的测试文件与用例，但不运行它们。它接收与 `run` 相同的文件筛选 [`RunOptions`](#runoptions)，外加 list 专属选项；仅用于执行的字段（`update`、`bail`、`shard`）会被忽略。

`printLocation` 会自行开启 task-location 收集 —— 不设置它时，结果中的 `test.location` 为 `undefined`。

- **类型：**

```ts
function listTests(
  options?: ListCommandOptions & RunOptions,
): Promise<ListCommandResult[]>;

type ListCommandOptions = {
  filesOnly?: boolean;
  json?: boolean | string;
  includeSuites?: boolean;
  printLocation?: boolean;
  summary?: boolean;
};
```

- **示例：**

```ts
const list = await rstest.listTests({ printLocation: true });
for (const file of list) {
  console.log(file.testPath, file.tests.length);
}
```

### rstest.mergeReports

将磁盘上的 blob reports 合并为单个聚合报告。

- **类型：**

```ts
function mergeReports(options?: {
  path?: string;
  cleanup?: boolean;
}): Promise<void>;
```

- **示例：**

```ts
await rstest.mergeReports({ path: './.rstest-reports', cleanup: true });
```

### rstest.close

释放实例资源，并还原在创建阶段拍下的 `process.env` / `process.exitCode` 快照。每次 `run()` 都会自行构建并拆除其 worker pool 与 Rsbuild server，因此在正常路径上 `close()` 几乎不做什么，但仍应始终调用它以还原宿主状态。

- **类型：**

```ts
function close(): Promise<void>;
```

## RunOptions

用于 `run`、`watch` 和 `listTests` 的每次调用选项。它们对应 CLI 的位置参数加上每次运行的 flags。

```ts
interface RunOptions {
  filters?: string[];
  filterMode?: 'fuzzy' | 'exact';
  testNamePattern?: RegExp | string;
  related?: boolean;
  changed?: boolean | string;
  update?: boolean;
  bail?: number | boolean;
  shard?: string | { index: number; count: number };
  project?: string[];
  passWithNoTests?: boolean;
}
```

- `filters`：位置形式的测试文件筛选，按 `filterMode` 匹配。
- `filterMode`：`filters` 的匹配策略：`'fuzzy'`（默认；大小写不敏感的子串匹配）或 `'exact'`（归一化路径相等）。对 `related` / `changed` 无效，它们始终精确匹配。
- `testNamePattern`：仅运行完整名称匹配的测试。字符串会经 `new RegExp` 转换。
- `related`：将位置形式的 `filters` 视为源文件，只运行与它们相关的测试。
- `changed`：从 git 推导运行集合：`true` = 工作区 + 暂存区改动；字符串 = 一个 `since` ref。不能与位置形式的 `filters` 同时使用。
- `update`：更新过期的 snapshots。
- `bail`：在 N 个测试失败后停止运行（`0` / `false` = 运行全部）。
- `shard`：只运行文件的一个切片，形式为 `'<index>/<count>'`（从 1 开始）或 `{ index, count }`。
- `project`：只运行指定名称的 projects（支持 `*` 通配、`!` 取反）。
- `passWithNoTests`：将未匹配到任何文件的运行视为通过而非失败。

## runCli

从已解析的 CLI 参数一次性运行 Rstest。这是与 Jest 兼容的入口，类似 `@jest/core` 的 `runCLI(argv, projects)`。它接收完整的 CLI 选项面（[`RstestArgv`](#rstestargv)），并 resolve 出 [`TestRunResult`](#testrunresult)。

与 `createRstest` 不同，`runCli` 对齐 `rstest run` CLI 命令：它会从 `cwd` 自动发现 config 文件并套用 CLI 默认值。它只运行一次。若需反复运行、检查 config、列举、合并报告或 watch 模式，请使用 [`createRstest`](#createrstest)。

- **类型：**

```ts
function runCli(
  argv?: RstestArgv,
  options?: { cwd?: string },
): Promise<TestRunResult>;
```

- **示例：**

```ts
import { runCli } from '@rstest/core/api';

// 位置形式的源文件位于 `_`，而 `related` 会把它们重新解读为源文件以解析相关测试。
const result = await runCli({ _: ['src/math.ts'], related: true }, { cwd });

console.log(result.ok, result.stats.tests);
```

### RstestArgv

`runCli` 接收的已解析 CLI 参数。它对应解析后的 `rstest <args>` 所产生的形态：命名 flags 加上位于 `_` 的位置文件。这是完整的 CLI 选项面（类似 Jest 的 `Config.Argv`），而 `RunOptions` 则是更小的每次运行子集。

```ts
type RstestArgv = CommonOptions & {
  // 位置参数。默认为测试文件筛选；当设置了 `related` / `findRelatedTests`
  // 时被视为源文件。
  _?: (string | number)[];
};
```

## TestRunResult

`rstest.run` 与 `runCli` resolve 出的值。

```ts
interface TestRunResult {
  ok: boolean;
  files: TestFileResult[];
  stats: {
    tests: {
      total: number;
      passed: number;
      failed: number;
      skipped: number;
      todo: number;
    };
    files: {
      total: number;
      failed: number;
    };
  };
  unhandledErrors: SerializedError[];
  duration: { total: number };
  snapshot?: SnapshotSummary;
  coverage?: CoverageMapData;
}
```

- `ok`：当没有测试、没有文件失败且没有 unhandled error 时为 `true`。当未匹配到任何测试文件时也为 `false`，除非设置了 `passWithNoTests`。它对齐 CLI 的 exit code。
- `files`：按文件聚合的结果（见 [`TestFileResult`](#testfileresult)）。
- `stats`：预先计算好的测试与文件计数。
- `unhandledErrors`：无法归因到单个测试的错误（worker 崩溃、config 加载失败）。
- `duration`：以毫秒计的墙钟耗时。
- `snapshot`：snapshot 汇总。仅当运行在任何测试执行前就中止时（例如 config 加载错误）才缺失。
- `coverage`：覆盖率数据。仅当解析后的 config 中 `coverage.enabled` 时存在。

### TestFileResult

按文件的结果。在 [`TestResult`](#testresult) 基础上扩展出逐用例的明细。

```ts
interface TestFileResult extends TestResult {
  // 该文件中发现的测试用例的扁平列表。
  results: TestResult[];
}
```

### TestResult

公开的单测试结果。它是 Rstest 内部 reporter 类型的稳定子集，因此内部重构不会破坏这一面。

```ts
interface TestResult {
  status: TestResultStatus;
  name: string;
  testPath: string;
  parentNames?: string[];
  duration?: number;
  errors?: SerializedError[];
  retryErrors?: SerializedError[];
  retryCount?: number;
  project: string;
}
```

- `status`：测试用例的最终状态。
- `name`：测试用例名（不含父级 suite 名）。
- `testPath`：测试文件的绝对路径。
- `parentNames`：父级 `describe` 块的名称，最外层在前。
- `duration`：以毫秒计的墙钟耗时；对 skipped / todo 测试缺失。
- `errors`：最后一次尝试产生的错误。
- `retryErrors`：配置了 `retry` 时，此前失败尝试产生的错误。
- `retryCount`：已执行的重试次数（首次尝试即通过时为 `0`）。
- `project`：来自 `projects` config 的 project 名；默认 project 为 `'default'`。

### SerializedError

一个可序列化的普通错误形态，贯穿于结果类型中使用（workers 运行在独立进程，因此错误会跨越进程边界）。

```ts
interface SerializedError {
  name: string;
  message: string;
  stack?: string;
  diff?: string;
  actual?: string;
  expected?: string;
  retryCount?: number;
  cause?: SerializedError;
}
```
