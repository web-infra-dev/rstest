# Programmatic API

<ApiMeta addedVersion="0.11.0" />

Besides the CLI, Rstest can be driven from your own Node.js code through the `@rstest/core/api` subpath â€” for build tools, custom runners, editor integrations, or scripts.

```ts
import { createRstest, runCli } from '@rstest/core/api';
```

`@rstest/core/api` is separate from the main `@rstest/core` entry. The main entry exposes the in-test runtime globals (`test`, `expect`, `rs`, â€¦), config helpers (`defineConfig`, `loadConfig`, `merge*`), and types; the programmatic runners live under `@rstest/core/api`.

:::warning Experimental
The programmatic API is experimental and its shape may change until `1.0.0`. Pin an exact `@rstest/core` version if you depend on it.
:::

Rstest offers two programmatic entries:

- [`createRstest`](#createrstest) â€” an async factory that returns a reusable instance. Use it to resolve config once and then run, list, watch, or merge repeatedly, or to inspect the resolved context.
- [`runCli`](#runcli) â€” a one-shot runner compatible with Jest's `runCLI`. Use it when you already have argv-shaped options, such as forwarding a parsed command line.

Both are host-safe: they never call `process.exit`, and they snapshot and restore `process.env` and `process.exitCode`, so a run cannot leave your process stuck in test mode or marked as failed.

## createRstest

Create a programmatic Rstest instance.

- **Type:**

```ts
function createRstest(options?: CreateRstestOptions): Promise<RstestInstance>;
```

Config is resolved during `createRstest`, so a config-load error is thrown here rather than on the first `run()`.

- **Example:**

```ts
import { createRstest } from '@rstest/core/api';

const rstest = await createRstest({
  cwd: './my-project',
  config: {
    include: ['**/*.test.ts'],
    reporters: [],
  },
});

const result = await rstest.run();
console.log(result.ok, result.stats.tests);

await rstest.close();
```

### Options

The first argument is an `options` object with the following properties:

```ts
interface CreateRstestOptions {
  cwd?: string;
  configFile?: string;
  config?: RstestConfig | RstestConfigFn;
  embedded?: boolean;
}
```

- `cwd`: Working directory. Defaults to `process.cwd()`, and is the base for resolving `root`.
- `configFile`: Path to a config file, relative to `cwd` or absolute. Unlike the CLI, the programmatic API does not auto-discover a config file from `cwd`; pass an explicit path, or omit it to run with `config` only.
- `config`: Inline configuration applied on top of `configFile`, or used alone. This is the single override channel for all config content, including `projects` and `reporters`.
  - **Object**: deep-merged over the disk config. Inline values win for scalars; arrays follow [`mergeRstestConfig`](/api/javascript-api/rstest-core#mergerstestconfig) semantics.
  - **Function** (`RstestConfigFn`): receives the disk-loaded config (an empty object when no `configFile` is set) and returns the final config, so you can transform it directly instead of merging an override on top.
- `embedded`: Defaults to `true` for the programmatic API, so a run cannot kill the host process. When `false`, Rstest installs `process.on('exit' | 'SIG*')` handlers and calls `process.exit()` on config errors, matching the CLI behavior.

### With a config file

Point at a config file on disk. `config` then becomes an optional override on top of it:

```ts
const rstest = await createRstest({
  cwd: './my-project',
  configFile: 'rstest.config.ts',
  config: {
    // overrides whatever the disk config declared
    testTimeout: 10000,
  },
});
```

### With a config function

The function form receives the resolved disk config and returns the final config, so you can transform it directly instead of merging an override object over it:

```ts
const rstest = await createRstest({
  cwd: './my-project',
  configFile: 'rstest.config.ts',
  config: (loaded) => ({
    ...loaded,
    // only run one of the files the disk config included
    include: ['a.test.ts'],
  }),
});
```

## RstestInstance

The object returned by `createRstest`. It holds the resolved config and runs tests against it on each call. It exposes the following members.

### rstest.context

The resolved context for the most recent build: the config, projects, and command. It is available as soon as `createRstest` resolves, so you can inspect the configuration without running any tests.

- **Type:**

```ts
readonly context: RstestContext;
```

### rstest.run

Run tests once and resolve a structured [`TestRunResult`](#testrunresult).

- **Type:**

```ts
function run(options?: RunOptions): Promise<TestRunResult>;
```

- **Example:**

```ts
const result = await rstest.run({
  filters: ['math'],
  testNamePattern: 'adds',
  bail: 1,
});
```

### rstest.watch

Start a watch session: run the matching tests, then re-run them as files change. Each run reports through the configured reporters rather than the resolved value; the promise resolves once watching has started. Call `close()` on the returned handle to stop watching and release the dev server and worker pool.

- **Type:**

```ts
function watch(options?: RunOptions): Promise<RstestWatcher>;
```

- **Example:**

```ts
const watcher = await rstest.watch({ filters: ['math'] });
// â€¦ later â€¦
await watcher.close();
```

### rstest.listTests

Collect matching test files and cases without running them. It accepts the same file-selection [`RunOptions`](#runoptions) as `run`, plus list-specific options; the execution-only fields (`update`, `bail`, `shard`) are ignored.

`printLocation` enables task-location collection on its own â€” without it, `test.location` in the results is `undefined`.

- **Type:**

```ts
function listTests(
  options?: ListCommandOptions & RunOptions,
): Promise<ListCommandResult[]>;

type ListCommandOptions = {
  filesOnly?: boolean;
  json?: boolean | string;
  includeSuites?: boolean;
  printLocation?: boolean;
  summary?: boolean;
};
```

- **Example:**

```ts
const list = await rstest.listTests({ printLocation: true });
for (const file of list) {
  console.log(file.testPath, file.tests.length);
}
```

### rstest.mergeReports

Merge on-disk blob reports into a single aggregate report.

- **Type:**

```ts
function mergeReports(options?: {
  path?: string;
  cleanup?: boolean;
}): Promise<void>;
```

- **Example:**

```ts
await rstest.mergeReports({ path: './.rstest-reports', cleanup: true });
```

### rstest.close

Release instance resources and restore the `process.env` / `process.exitCode` snapshot taken at creation. Each `run()` already builds and tears down its own worker pool and Rsbuild server, so `close()` does little on the current happy path, but you should always call it to restore host state.

- **Type:**

```ts
function close(): Promise<void>;
```

## RunOptions

Per-invocation options for `run`, `watch`, and `listTests`. They map to the CLI's positional arguments plus per-run flags.

```ts
interface RunOptions {
  filters?: string[];
  filterMode?: 'fuzzy' | 'exact';
  testNamePattern?: RegExp | string;
  related?: boolean;
  changed?: boolean | string;
  update?: boolean;
  bail?: number | boolean;
  shard?: string | { index: number; count: number };
  project?: string[];
  passWithNoTests?: boolean;
}
```

- `filters`: Positional test-file filters, matched per `filterMode`.
- `filterMode`: Matching strategy for `filters`: `'fuzzy'` (default; case-insensitive substring) or `'exact'` (normalized path equality). Ignored for `related` / `changed`, which always match exactly.
- `testNamePattern`: Run only tests whose full name matches. A string is coerced via `new RegExp`.
- `related`: Treat positional `filters` as source files and run only their related tests.
- `changed`: Derive the run set from git: `true` = working-tree + staged changes; a string = a `since` ref. Cannot be combined with positional `filters`.
- `update`: Update outdated snapshots.
- `bail`: Stop the run after N failing tests (`0` / `false` = run all).
- `shard`: Run only a slice of files, as `'<index>/<count>'` (1-based) or `{ index, count }`.
- `project`: Run only the named projects (`*` wildcards, `!` negation).
- `passWithNoTests`: Treat a run that matched no files as a pass instead of a failure.

## runCli

Run Rstest once from parsed CLI options. This is the Jest-compatible entry, similar to `@jest/core`'s `runCLI(argv, projects)`. It accepts the full CLI option surface ([`RstestArgv`](#rstestargv)) and resolves to a [`TestRunResult`](#testrunresult).

Unlike `createRstest`, `runCli` matches the `rstest run` CLI command: it auto-discovers the config file from `cwd` and applies the CLI defaults. It runs once. For repeated runs, config inspection, listing, report merging, or watch mode, use [`createRstest`](#createrstest).

- **Type:**

```ts
function runCli(
  argv?: RstestArgv,
  options?: { cwd?: string },
): Promise<TestRunResult>;
```

- **Example:**

```ts
import { runCli } from '@rstest/core/api';

// Positional source files live in `_`, and `related` reinterprets them as
// source files to resolve related tests.
const result = await runCli({ _: ['src/math.ts'], related: true }, { cwd });

console.log(result.ok, result.stats.tests);
```

### RstestArgv

The parsed CLI options accepted by `runCli`. It matches what a parsed `rstest <args>` produces: named flags plus positional files in `_`. This is the full CLI option surface (similar to Jest's `Config.Argv`), whereas `RunOptions` is a smaller per-run subset.

```ts
type RstestArgv = CommonOptions & {
  // Positional arguments. Test-file filters by default; treated as source
  // files when `related` / `findRelatedTests` is set.
  _?: (string | number)[];
};
```

## TestRunResult

The value resolved by `rstest.run` and `runCli`.

```ts
interface TestRunResult {
  ok: boolean;
  files: TestFileResult[];
  stats: {
    tests: {
      total: number;
      passed: number;
      failed: number;
      skipped: number;
      todo: number;
    };
    files: {
      total: number;
      failed: number;
    };
  };
  unhandledErrors: SerializedError[];
  duration: { total: number };
  snapshot?: SnapshotSummary;
  coverage?: CoverageMapData;
}
```

- `ok`: `true` when no test and no file failed and there were no unhandled errors. It is also `false` when no test file matched, unless `passWithNoTests` is set. Mirrors the CLI exit code.
- `files`: Per-file aggregate results (see [`TestFileResult`](#testfileresult)).
- `stats`: Pre-computed counts for tests and files.
- `unhandledErrors`: Errors not attributable to a single test (worker crash, config load failure).
- `duration`: Wall-clock duration in ms.
- `snapshot`: Snapshot summary. Absent only when the run aborted before any test executed (for example, a config load error).
- `coverage`: Coverage data. Present only when `coverage.enabled` in the resolved config.

### TestFileResult

Per-file result. Extends [`TestResult`](#testresult) with the per-case breakdown.

```ts
interface TestFileResult extends TestResult {
  // Flattened list of test cases discovered in this file.
  results: TestResult[];
}
```

### TestResult

Public per-test result. It is a stable subset of Rstest's internal reporter type, so internal refactors do not break this surface.

```ts
interface TestResult {
  status: TestResultStatus;
  name: string;
  testPath: string;
  parentNames?: string[];
  duration?: number;
  errors?: SerializedError[];
  retryErrors?: SerializedError[];
  retryCount?: number;
  project: string;
}
```

- `status`: Final state of the test case.
- `name`: Test case name (no parent suite names included).
- `testPath`: Absolute path to the test file.
- `parentNames`: Names of parent `describe` blocks, outermost first.
- `duration`: Wall-clock duration in ms; absent for skipped / todo tests.
- `errors`: Errors from the final attempt.
- `retryErrors`: Errors from previous failed attempts when `retry` is configured.
- `retryCount`: Number of retries performed (`0` when the first attempt passed).
- `project`: Project name from `projects` config; the default project is `'default'`.

### SerializedError

A plain, serializable error shape used throughout the result types (workers run in separate processes, so errors cross a process boundary).

```ts
interface SerializedError {
  name: string;
  message: string;
  stack?: string;
  diff?: string;
  actual?: string;
  expected?: string;
  retryCount?: number;
  cause?: SerializedError;
}
```
